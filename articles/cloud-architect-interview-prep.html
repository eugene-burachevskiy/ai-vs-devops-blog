🎯 0. FRAMING — What This Role Actually Is

This is NOT a build/configure DevOps role. It is a consulting/architecture review role. You will:

  • Read existing architecture docs, DR runbooks, and test evidence
  • Critically assess resilience across GKE, AKS, Istio, TIBCO, APIM, Azure DevOps, Cognitive Services, Ratabase

  • Write defensible findings for a board-level (FTSE100) audience

The interviewer is testing: Can you think like an architect who has operated these systems, not just read about them? Can you critically assess, not just configure? Do you understand DR trade-offs (RTO/RPO, cost, complexity)?

🛡️ 1. Disaster Recovery Fundamentals

Q1.1 — How would you structure a DR review/assessment for a cloud estate?

Strong answer:

  1. Scope & tiering — Inventory all ~300 apps/services, classify into DR tiers by business criticality. Not everything gets active-active.
  2. Define metrics per tier — RTO (Recovery Time Objective = how fast back), RPO (Recovery Point Objective = how much data loss tolerable). Insurance claims/policy = RTO minutes; analytics = RTO days.
  3. Current-state review — Architecture diagrams, existing DR runbooks, last test evidence, dependency maps. Gap: what's undocumented, untested.
  4. Dependency mapping — Upstream/downstream for each critical app. Single points of failure. Cascading failure paths.
  5. Scenario assessment — Walk through: AZ loss, region loss, service outage, dependency outage (e.g., TIBCO down → what breaks?).
  6. Identify gaps — Missing runbooks, untested failover, stale backups, missing cross-region replication, RPO violations.
  7. Recommendations — Prioritized, cost-aware, with remediation effort and risk rating.
  8. Test strategy — DR drill cadence (game days), chaos engineering, evidence capture.

Keywords: RTO/RPO, DR tiers, dependency mapping, SPOF, blast radius, recovery walkthrough, cascading failure.

Q1.2 — RTO vs RPO: how do they drive architecture?

  • RPO: Max acceptable data loss (time). Drives backup frequency / replication. RPO=0 → synchronous replication. RPO=15min → async acceptable.
  • RTO: Max acceptable downtime (time). Drives recovery automation. RTO=minutes → active-active. RTO=hours → automated failover. RTO=days → restore from backup.
  • Trade-off: RTO/RPO→0 = high cost & complexity. Tiering lets you spend where it matters.

Example: Insurer claims system: RTO 4h, RPO 15min (regulatory). Analytics warehouse: RTO 24h, RPO 24h (fine).

🪨 2. Kubernetes Resilience — GKE & AKS

Q2.1 — Design a HA Kubernetes cluster. GKE vs AKS specifics?

Control plane:

  • GKE: Regional (multi-zone) control plane — 3 zones free. Zonal = single master = SPOF. Always regional for prod.
  • AKS: Free tier = single no-SLA master. For prod, use Availability Zones (3 AZs) + Standard (VM-backed) or Premium (99.9% SLA) tier.

Worker nodes:

  • Spread across 3+ AZs with separate node pools (system pool, user pool, spot pool).
  • PodDisruptionBudget (PDB) for voluntary evictions; topologySpreadConstraints or podAntiAffinity to distribute across zones/nodes.
  • Cluster autoscaler + HPA. Graceful shutdown: terminationGracePeriodSeconds, preStop hooks.

Q2.2 — What happens when a node fails? Walk through recovery.

  1. Detection: Node controller marks NotReady after node-monitor-grace-period (default 40s).
  2. Pod eviction: After pod-eviction-timeout (default 5m), pods scheduled for deletion & reschedule.
  3. Stateful workloads: Pod won't reschedule until PVC reattaches. Zone-bound PVs prevent cross-zone failover — this is a common DR gap.

⚠️ Critical DR insight: Stateful workloads are hardest. Node loss → pod reschedule → PVC stuck if disk is zone-bound. Regional PDs (GKE: 2 zones) / zone-redundant storage (Azure ZRS) are essential for cross-zone failover.

Q2.3 — Can a Kubernetes cluster survive region loss?

  • Single cluster cannot survive region loss by itself.
  • Multi-cluster, multi-region — Clusters in 2+ regions, global LB (GCP: Global HTTP(S) LB; Azure: Front Door).
  • Cluster rebuild — GitOps (ArgoCD) rebuilds cluster from scratch in DR region. RTO = provision + restore data time.
  • Backup: Velero backs up cluster resources + PVs to S3/GCS/Azure Storage; restore into new cluster.
  • Stateful data: Replicate databases separately (read replicas, cross-region, PITR).

🔴 3. Istio Service Mesh — HARD REQUIREMENT

⚠️ This is a hard requirement. Expect 3–5 questions. Frame everything through DR/resilience lens.

Q3.1 — Istio components & SPOFs

  • Data plane: Envoy proxies (sidecars) — handles mTLS, routing, retries, circuit breaking, telemetry.
  • Control plane: Istiod (merged Pilot+Citadel+Galley since 1.5) — pushes config to Envoy, manages certs.

SPOFs:

  • Istiod — horizontally scalable. If down, existing Envoy config persists (eventually consistent). BUT no new config pushes, and certs expire (default 24h TTL, rotate at 80%). If Istiod is down past expiry → mTLS breaks → mesh traffic fails. Critical DR scenario.
  • Envoy sidecars — per-pod; sidecar failure = that pod's traffic fails.
  • Ingress Gateway — multiple replicas across zones behind LB.
  • Root CA — if lost, all certs invalid; need CA backup/DR.

Q3.2 — mTLS in Istio: how to assess from DR perspective?

  • Identity: SPIFFE-based (SPIRE), derived from K8s service account.
  • Modes: DISABLE, SIMPLE (one-way), MUTUAL (mTLS). Permissive (accepts both) for migration; Strict for prod.
  • Policy: PeerAuthentication (mTLS per namespace/workload), RequestAuthentication (JWT).

DR questions an architect asks:

  • Is mTLS Strict everywhere? Permissive gaps = migration debt, security DR gap.
  • What's cert TTL and rotation? If control plane down, how long until certs expire?
  • Is the root CA backed up? Custom CA or Istio self-signed?
  • Partial mesh failure plan (some sidecars up, some down)?
  • Cross-cluster mTLS — shared root CA required; how distributed/backed up?
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: STRICT

Q3.3 — VirtualServices & DestinationRules for resilience

VirtualService — how to route (L7):

  • Canary, blue-green, A/B via weights. Traffic shifting for failover.
  • Retries: retries: 3, perTryTimeout: 2s, retryOn: 5xx,reset
  • Timeouts prevent cascading failures. Fault injection for chaos testing.

DestinationRule — to whom (subsets + traffic policy):

  • Subsets (e.g., v1, v2; region-a, region-b) → enables cross-region failover.
  • Circuit breaking — connection pools, outlier detection (eject unhealthy).
  • Locality LB — prefer local zone, failover to other zones/regions.
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
spec:
  http:
  - route:
    - destination:
        host: payments.local
    retries:
      attempts: 3
      perTryTimeout: 2s
      retryOn: 5xx,reset,connect-refused,refused-stream
    timeout: 10s
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
spec:
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        http1MaxPendingRequests: 50
        maxRequestsPerConnection: 10
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50

Q3.4 — Circuit breaking: how and why?

Outlier detection: Monitors each endpoint; N consecutive 5xx → eject for baseEjectionTime. Up to maxEjectionPercent ejected. Auto-reintroduced.

Connection pools: maxConnections, maxPendingRequests prevent overload cascade. Limits hit → requests queue or fail fast.

Why: Without circuit breaking, slow downstream backs up connections → cascading failure → total outage. CBR contains blast radius. Core to resilience.

Q3.5 — Istio control-plane DR runbook

Assessment checklist:

  1. Istiod replicas — multiple across AZs? HPA?
  2. Resources adequate for mesh size?
  3. Certificate management — TTL, rotation window, CA backup. If certs expire, mesh dies.
  4. Config backup — CRDs in Git (GitOps)?
  5. Version & upgrades — supported? Istio has aggressive deprecation.
  6. Sidecar resource limits — undersized = latency/failure under load.
  7. Observability — Kiali, Prometheus for mTLS health, config push success, sidecar drift.

Istiod loss runbook:

  1. Detect (alerts on Istiod down).
  2. Restore Istiod from Git (Helm/istioctl).
  3. Verify sidecars retain last-good config (steady-state OK).
  4. Verify cert rotation resumes before expiry.
  5. Test config push, mTLS health.

Q3.6 — Multi-cluster Istio for DR

  • Single mesh, multiple clusters — shared root CA, cross-cluster service discovery. Failover via DestinationRule localityLbSetting.
  • Primary-remote: One Istiod controls all — simpler, single control plane.
  • Multi-primary: Each cluster has Istiod; shared root CA, synced service registry. Most resilient.
  • Gotcha: Shared root CA must be backed up. Cross-cluster service registry sync must be monitored.

Ref: istio.io/latest/docs/setup/install/multicluster/

🔰 4. Azure Platform DR

Q4.1 — Design for Azure regional failover

  • Region pairs: Each Azure region has a paired region (e.g., North Europe ↔ West Europe). Prioritized recovery, sequential updates, 300-mile separation.
  • Azure Site Recovery (ASR): VM replication to paired region; automated failover with runbooks.
  • Front Door: Global L7 LB + WAF + URL routing. Multi-region active-active/active-passive with health probes.
  • Traffic Manager: DNS-based distribution (priority, weighted, performance, geographic).
  • For AKS: Clusters in paired regions; Front Door routes; state via Azure Storage GRS, Cosmos DB multi-region, SQL DB geo-replication.

Q4.2 — Azure APIM resilience & DR

  • Multi-region: APIM Premium supports multi-region gateway deployment; management plane in primary.
  • Failover: Traffic Manager/Front Door routes to healthy regional gateway.
  • Config DR: Config stored in primary region. Back up via API/ARM templates/Git. Premium supports self-hosted gateway (containerized, deploy to AKS).
  • Common gaps: Config only in primary, no backup. Secrets not replicated (Key Vault dependency). Single-region. No tested failover runbook.

Q4.3 — Azure DevOps resilience

  • Managed SaaS; Microsoft handles HA. But org region matters — if West Europe is down, you can't deploy.
  • Risk: Self-hosted agents in failed region → pipelines fail. Artifact storage region-bound → deployments blocked.
  • Mitigations: Deploy agents across regions. Mirror artifacts externally (JFrog). Backup repos to GitHub/Storage. Pipeline YAML is Git-backed (good).
  • Common gap: CI/CD capability recovery undocumented in DR plan.

Q4.4 — Azure Backup vs ASR

  • Azure Backup: Data backup (VMs, files, SQL) to Recovery Services Vault. RPO-driven schedule. RTO = restore time. For data protection, ransomware, accidental deletion.
  • ASR: Continuous VM replication to paired region; orchestrated failover. Lower RTO/RPO. For regional failover.
  • Both: Backup for granular PITR; ASR for regional failover.
  • Containers: Neither backs up AKS state natively — use Velero.

Q4.5 — Key Vault DR

  • Soft delete (mandatory): 7–90 day retention. Purge protection prevents hard delete.
  • No native cross-region replication. For DR: replicate secrets to paired region vault (automated scripting); or use managed identities + IaC to recreate.
  • Assessment: If Key Vault is gone and AKS in DR region can't get secrets → app fails. Are secrets recoverable?

☘️ 5. GCP Platform DR

Q5.1 — Global vs regional load balancing for DR

  • Global External HTTP(S) LB: Anycast IP, single global IP, routes to nearest healthy backend across regions. Best for multi-region active-active. Health checks auto-remove failed regions.
  • Cloud CDN for static content. Cloud Armor — WAF/DDoS, attaches to global LB.
  • DR pattern: Global LB + backends in 2+ regions → automatic regional failover at LB layer. Core GCP multi-region DR.
  • Regional LB: Single region only. Use Traffic Director for multi-region internal LB.

Q5.2 — GCS: region vs dual-region vs multi-region

  • Region bucket: Lowest latency in-region; SPOF for region loss.
  • Dual-region: 99.99% availability, geo-redundant across 2 regions. Best for DR-sensitive data.
  • Multi-region: 99.95–99.99%, geo-redundant across large geo (US, EU).
  • Object Versioning + Lifecycle protects from accidental deletion.
  • For DR: Backups (Velero, DB snapshots) use dual-region or cross-region replication.

Q5.3 — Cloud Armor in DR context

  • WAF (SQLi, XSS, L7 filter, geo-based, IP-based) + DDoS for GCP LBs.
  • Standard vs Managed Protection Plus: Managed Plus for advanced DDoS, adaptive protection.
  • DR fit: Prevents traffic-based incidents (DDoS, app-layer attacks); enables origin failover via LB backend policies.

🕸️ 7. Dependency Mapping & Failure Propagation (~300 apps)

Q7.1 — Approach for mapping dependencies across ~300 apps

Methodology:

  1. Inventory — CMDB, ServiceNow, Dynatrace/Datadog service map.
  2. Taxonomy — Apps, services, data stores, shared platforms (TIBCO, APIM, DBs), external deps.
  3. Top-down: Start from business processes (e.g., "file a claim"), trace through apps to infra. Prioritizes by business criticality.
  4. Bottom-up: Service mesh telemetry (Istio Kiali), APM (Dynatrace), VPC flow logs, DNS queries. Reveals actual vs documented dependencies.
  5. Visualization: Graph (Kiali, Graphviz, Neo4j). Critical path identification — shared deps = high blast radius.
  6. Document failure scenarios: "TIBCO down" → list impacted apps; "Region A lost" → impacted services.

Red flags: Undocumented deps, shared SPOFs (one DB for 50 apps), circular dependencies, hidden external third-party APIs.

Ref: TOGAF/ArchiMate notation; Istio Kiali, Dynatrace, Datadog Service Map, ServiceNow CMDB.

Q7.2 — Blast radius: what is it and how to reduce?

  • Blast radius: Extent of impact when a component fails. Large = single component takes down many services.
  • Reduced by: Decoupling (async messaging, CBR), redundancy (multi-AZ/region), isolation (bulkheads), graceful degradation (fallbacks), backpressure (rate limiting).
  • In mesh: Istio CBR + outlier detection contain failures. Without → cascade.
  • In data: Separate DBs per service (no shared DB), read replicas.

🗄️ 8. Database DR (Platform Context)

JD says you provide platform context, not DB engineering. Focus on platform-level assessment.

Q8.1 — PostgreSQL DR on GKE (Ratabase context)

  • Cloud SQL for PostgreSQL: HA config (regional, sync standby in another zone, auto-failover). Cross-region read replica for DR.
  • Self-managed on GKE (Ratabase case) — bigger DR risk. You manage replication, backups, failover.
  • Assess: Backup strategy (pg_dump vs WAL archiving vs pgBackRest for PITR). Streaming replica (where? cross-zone? cross-region?). Failover automation (Patroni, Stolon). PV type (regional PD?).
  • PITR: WAL archiving to object storage → restore to any point.
  • RPO/RTO: Sync replica cross-zone = RPO 0, RTO minutes. Async cross-region = RPO seconds–min, RTO min–hours.

Q8.2 — Azure Cognitive Services resilience (~5 services)

  • Managed services — Microsoft handles intra-region HA.
  • DR: Resource is regional. Redeploy in paired region (IaC). Custom data/models must be replicated/backed up.
  • Multi-region: Deploy same services in paired region; route via Front Door. Active-active or active-passive.

🔍 9. Monitoring & Observability for DR

Q9.1 — What observability is critical for DR readiness?

  • Health probes — LB/APIM/Gateway health checks for regional/service failure detection.
  • Synthetic monitoring (uptime checks) — proactive external detection.
  • SLI/SLO dashboards — know when error budget burns.
  • DR-specific: Replication lag (DB), cert expiry (Istio mTLS), backup success, failover runbook test results.
  • Alerting: Multi-channel (PagerDuty), escalation, runbook links.

Assessment: Is observability sufficient to detect the failure scenarios in the DR plan? Many plans fail because alerts are noisy/ignored or too slow.

📝 10. Consulting & Delivery (Soft Skills)

Q10.1 — Present technical findings to a non-technical board?

  • Lead with business impact (revenue, regulatory, customer), not tech.
  • Risk rating (high/med/low) with quantified impact and concrete recommendations.
  • Visuals — diagrams, heat maps, dependency graphs.
  • Defensible — every finding backed by evidence (config, logs, test results).
  • Prioritized — group and rank, not a dump of 200 findings.

Q10.2 — Handle disagreement with a customer architect?

  • Evidence-based, not opinion. Reference industry standards (NIST, CIS, cloud best practices, PRA/FCA for UK insurance).
  • Acknowledge their constraints (cost, legacy, org context).
  • Frame as risk, not failure: "This is a risk because…" not "This is wrong."
  • Document formally if needed.

🎭 11. Scenarios You Should Practice Narrating

Practice walking through each end-to-end:

  1. "Azure region is lost" — Region pair failover, Front Door/Traffic Manager, AKS in DR (GitOps rebuild), DB geo-replication, Key Vault secrets in DR, APIM redeploy, DNS TTL.
  2. "GKE single zone fails" — Regional cluster (other zones serve), regional PDs, HPA reschedules, PDB protects quorum, Istio outlier detection ejects dead zone endpoints.
  3. "TIBCO is down" — Map dependent apps (from dependency map); async flows queue or fail; sync flows break → user impact. Recovery order: infra → TIBCO → flows.
  4. "Istiod is down" — Steady-state OK (Envoy has last config). But: no config pushes, certs expire (24h), no new routing. Restore from Git before expiry.
  5. "mTLS certs expiring during Istiod rebuild" — Race condition; mesh dies if certs expire before Istiod issues new. Mitigation: faster recovery, longer cert TTL, sidecar injection of new root CA.
  6. "PostgreSQL on GKE — ensure DR" — Regional PD, streaming replica cross-zone, WAL archiving to GCS dual-region, pgBackRest, Patroni auto-failover, tested restore drills.
